    **CRITICAL FIX (RESEARCH.md Pitfall 2):** The product detail Blade view currently iterates technical_specs as a flat key-value object:
    ```blade
    @foreach($product->technical_specs as $spec => $value)
        <tr><td>{{ $spec }}</td><td>{{ $value }}</td></tr>
    @endforeach
    ```

    This must be changed to iterate as an array of objects to match the Filament Repeater output format (per D-08, D-09, and RESEARCH.md lines 420-438):

    ```blade
    @if($product->technical_specs && is_array($product->technical_specs) && count($product->technical_specs) > 0)
        @foreach($product->technical_specs as $spec)
            <tr>
                <td>{{ $spec['attribute_name'] }}</td>
                <td>{{ $spec['attribute_value'] }}</td>
            </tr>
        @endforeach
    @endif
    ```

    **Why this change is needed:** The Filament Repeater stores specs as `[{"attribute_name":"Công suất","attribute_value":"200W"}]` (array of objects). The old code expected `{"Công suất":"200W"}` (flat object). Without this fix, the spec table would render "attribute_name" as the key and the entire first spec object as the value.

    **Additional guard:** Add `@if($product->technical_specs && is_array($product->technical_specs) && count($product->technical_specs) > 0)` check to handle:
    - Null `technical_specs` (no specs entered)
    - Empty array `[]` (Repeater with no rows)
    - Any non-array value (edge case from data migration)

    Also verify that the gallery image paths work. The images are stored as array of filenames like `["products/uuid_image.jpg"]`. The view uses `{{ $product->images[0] ?? asset('img/product-1.jpg') }}` which will resolve to `storage/products/uuid_image.jpg`. Verify that the asset() helper or Storage::url() is used correctly — if the stored path is relative to storage/app/public, the view needs `Storage::url($image)` or `asset('storage/' . $image)`.

    **Fix image path rendering:** The images are stored by Filament FileUpload as relative paths like `products/uuid_filename.jpg` within the `public` disk (which maps to `storage/app/public/`). To serve them, the Blade view should use `asset('storage/' . $image)` instead of just `{{ $image }}`.

    Find all image references in the file (lines 92, 104, 150, 195, 215, 238) and wrap them:
    ```blade
    asset('storage/' . $product->images[0])
    ```
    (Only if the current code already uses `asset()` — inspect each usage. Line 92 uses `{{ $product->images[0] ?? asset('img/product-1.jpg') }}` — fix to `{{ $product->images[0] ? asset('storage/' . $product->images[0]) : asset('img/product-1.jpg') }}`)

    Also update the brochure link at line 150: `href="{{ asset($product->brochure) }}"` — if brochure is stored on `public` disk, it should be `asset('storage/' . $product->brochure)`.

    **Repeater stores array of objects but product seeder may store flat objects:**
    Verify with the ProductSeeder (Plan-02) that `technical_specs` is stored as array-of-objects in the seeder too, to maintain consistency. The ProductSeeder was instructed to use `[['attribute_name' => '...', 'attribute_value' => '...']]` format.
